# Email Handling Agent — Implementation Plan

## Overview

An email processing agent that ingests Outlook messages via a shared directory, classifies and routes them using LLM reasoning, notifies the user via configurable channels, and emits a YAML command file that a Windows-side PowerShell script executes to mutate Outlook (move, flag, draft).

The agent **never runs PowerShell directly** — it runs in a Linux container. All Outlook interaction happens through files in a shared directory.

---

## Architecture Diagram

```
  Windows Host                         Linux Container (Council)
  ─────────────────────────────        ────────────────────────────────────
  GetOutlook.ps1                        data/messages/email/inbox/
    │  emits one YAML per email  ──►    *.yaml  (standard mailbox format)
    │
  Task Scheduler (periodic)
    │
  ExecuteOutlookCommands.ps1   ◄──     data/workspace/outlook/commands/
    │  reads commands.yaml              commands.yaml  (written by agent)
    │  dispatches to:
    ├─ ManipulateOutlook.ps1
    └─ DraftOutlook.ps1
```

## Shared Directory Layout

```
data/
  messages/
    email/
      inbox/        ← GetOutlook.ps1 drops one YAML per email here
      processed/    ← standard mailbox: mark_message_processed moves files here
  workspace/
    outlook/
      commands/     ← agent writes commands.yaml here; ExecuteOutlookCommands.ps1 reads + deletes it
```

## Component 1 — PowerShell Scripts (`scripts/outlook/`)

### 1.1 `GetOutlook.ps1`

Move from `experiments/`. Modify to write one YAML file per email into a target directory instead of emitting JSON to stdout. Each file follows the Council mailbox schema (so it is readable by `check_inbox` unchanged), with additional `outlook_*` fields for structured metadata.

**Output file format** — `data/messages/email/inbox/<msg_id>.yaml`:

```yaml
msg_id: a3f9c12e0b4d          # 12-char hex, generated by script
from_agent: outlook
from_session: outlook-20260420T083000Z
reply_to_session: null
prompt: |                     # formatted for LLM consumption
  Subject: Q1 Budget Review
  From: Alice Smith <alice@example.com>
  Received: 2026-04-20 08:28:00
  Folder: Inbox

  Please find attached the Q1 budget review...
sent_at: "2026-04-20T08:28:00+00:00"
# Outlook-specific fields (extra, ignored by standard mailbox tools):
outlook_entry_id: "000000..."
outlook_store_id: "000000..."
outlook_mailbox: "user@example.com"
outlook_folder: "Inbox"
outlook_subject: "Q1 Budget Review"
outlook_sender: "Alice Smith <alice@example.com>"
outlook_received: "2026-04-20T08:28:00"
outlook_body_html: "<html>..."  # optional, omitted if null
```

**Parameters:** `-MailboxName`, `-Folders` (array), `-Since` (e.g. `25h`), `-OutputDir` (path to inbox directory)

### 1.2 `ManipulateOutlook.ps1`

Accepts `-Action` (move | flag | unflag | read | unread), `-EntryId`, `-StoreId`, and `-DestinationFolder` (required for move).

### 1.3 `DraftOutlook.ps1`

Creates a message in the Drafts folder. Parameters: `-To`, `-Subject`, `-Body`, `-MailboxName`.

### 1.4 `ExecuteOutlookCommands.ps1`

- Reads `commands.yaml` from a configurable `-CommandsFile` path
- Dispatches each command to `ManipulateOutlook.ps1` or `DraftOutlook.ps1`
- Deletes (or archives) the commands file on completion
- Logs each action with result to stdout

**Command file format** (`data/workspace/outlook/commands/commands.yaml`):

```yaml
commands:
  - action: move
    entry_id: "000000..."
    store_id: "000000..."
    destination_folder: "Processed"

  - action: flag
    entry_id: "000000..."
    store_id: "000000..."

  - action: unflag
    entry_id: "000000..."
    store_id: "000000..."

  - action: read         # mark as read
    entry_id: "000000..."
    store_id: "000000..."

  - action: unread
    entry_id: "000000..."
    store_id: "000000..."

  - action: draft
    mailbox: "user@example.com"
    to: "person@example.com"
    subject: "Re: Q1 Budget Review"
    body: "Hi Alice,\n\nThank you for sending this over..."
```

**Windows Task Scheduler Setup** (manual, documented):
- Program: `powershell.exe`
- Arguments: `-NonInteractive -File "C:\path\to\ExecuteOutlookCommands.ps1" -CommandsFile "D:\dev\Council\data\workspace\outlook\commands\commands.yaml"`
- Trigger: e.g. every 15 minutes, or on file-creation event for `commands.yaml`

## Component 2 — Python Tools (`tools/outlook_tools.py`)

Three tools, all pure file I/O — no subprocess, no Outlook COM access.

### `write_outlook_commands(commands: list[dict]) -> str`

Serialises the agent's intended Outlook actions as YAML to `data/workspace/outlook/commands/commands.yaml`. Returns a confirmation string. Overwrites any existing file (the agent should consolidate all actions into one call at end of session).

### `lookup_agent_directory() -> str`

Reads all `agents/*.yaml` files and returns a YAML-formatted list of agent names and their descriptions. Used by the LLM to decide which agent to route an email to.

### `notify(provider: str, destination: str, content: str, context: ToolContext) -> str`

Posts a message to a configured channel. `provider` and `destination` come from the agent's YAML config. Dispatch:

```python
if provider == "discord":
    return post_to_discord_channel(destination, content, context)
# elif provider == "slack": ...
```

No credential management — reuses existing tool implementations which already read from environment variables.

## Component 3 — Email Agent

### `agents/email.yaml`

```yaml
agent_id: email
name: Email Agent
description: >
  Processes incoming Outlook emails: classifies, routes to agents,
  notifies the user, and queues Outlook actions (move, flag, draft replies).

tools:
  - check_inbox
  - mark_message_processed
  - send_message        # for routing to other agents
  - write_outlook_commands
  - lookup_agent_directory
  - notify

permissions:
  allowed_paths:
    - data/workspace/outlook
    - agents

notify_channels:
  - provider: discord
    destination: "123456789012345678"   # channel ID, fill in at deploy time
```

### `flows/email_loop.yaml`

```
Step 1 — Check inbox
  check_inbox → if empty, end session

Step 2 — Classify emails
  For each message in inbox:
    LLM classifies as one of:
      - route_to_agent: forward to another agent's inbox
      - notify_user: include in summary only
      - action_needed: flag + draft reply + notify user
      - ignore: mark processed, no further action

Step 3 — Route emails
  For each route_to_agent email:
    lookup_agent_directory (once, cached in state)
    LLM picks target agent
    send_message → target agent inbox
    (email body becomes the message prompt; outlook_* fields preserved)

Step 4 — Emit notifications
  Compose summary (counts + subject lines of notify_user and action_needed items)
  For each channel in notify_channels:
    notify(provider, destination, summary)

Step 5 — Write Outlook commands
  Consolidate all intended moves, flags, drafts into one list
  write_outlook_commands(commands)

Step 6 — Mark all processed
  mark_message_processed for each handled inbox message
```

## Implementation Order

The tasks within each numbered group can be done in parallel; groups are sequential.

```
Group 1 (parallel):
  1a. GetOutlook.ps1 — update to write YAML mailbox files
  1b. ManipulateOutlook.ps1 — new script
  1c. DraftOutlook.ps1 — new script

Group 2 (depends on 1):
  2a. ExecuteOutlookCommands.ps1 — dispatches to 1b + 1c

Group 3 (parallel, independent of Group 1/2):
  3a. tools/outlook_tools.py — write_outlook_commands, lookup_agent_directory, notify
  3b. tests/test_outlook_tools.py — unit tests (file I/O mocked)

Group 4 (depends on 3):
  4a. agents/email.yaml
  4b. flows/email_loop.yaml

Group 5 (depends on all):
  5a. Integration test: place synthetic YAML in inbox, run flow, assert commands.yaml produced
  5b. Document Task Scheduler setup in README or ops runbook
```

## Open Questions / Not in Scope

- Attachment handling: out of scope for now; `bodyHtml` field preserved but agents treat body as text
- Sending email directly (non-draft): deliberately excluded; all sends go through Drafts for human review
- Multiple Outlook mailboxes simultaneously: `GetOutlook.ps1` supports it via `-MailboxName`; agent treats all emails uniformly
